# MockDataGenerate 旧版实现

## 适用代码版本

本文说明当前工作区修改前的旧版 `_dev:mock-data:generate` 命令实现。

旧版依据：

- `HEAD:app/Console/Commands/Dev/MockDataGenerate.php`

旧版只有一个主要文件：

- `app/Console/Commands/Dev/MockDataGenerate.php`

## 命令入口

命令签名：

```bash
php artisan _dev:mock-data:generate --month=1-12
```

`--month` 接收月份区间。旧版同样按月份循环设置 `setting.gen.month.current`，再生成对应月份的演示数据。

该命令会清理并重建大量演示数据，不适合作为普通调试命令随意执行。

## 执行流程

旧版全部逻辑集中在 `MockDataGenerate::handle()` 中，整体流程为：

```text
清理业务数据
  -> 清理审计日志表
  -> 创建基础数据
  -> 按月份循环
      -> 随机选车辆和客户
      -> 创建车辆整备
      -> 创建合同
      -> 直接在循环内创建所有合同关联数据
```

前半段清理和基础数据创建与当前版本类似：

- 删除付款、车辆、合同、客户、普通管理员和相关业务数据。
- 删除 `setting.dblog.models` 中配置模型对应的审计表数据。
- 创建基础管理员、车型、车辆、门店和客户。
- 加载车辆、客户和启用状态门店。

差异主要在每个月的业务数据生成阶段：旧版没有独立的场景类，合同、付款、车辆状态、验车、维保、违章、结算、保险和年检都直接写在同一个循环里。

## 生成的数据范围

旧版会清理并重新生成以下核心数据：

- 付款相关：付款账户余额、付款流水、付款记录。
- 车辆相关：车辆、车型、整备、验车、行程、维修、保养、保险、年检、违章、事故、强制收车、门店。
- 租赁相关：销售合同、合同结算、临时换车。
- 客户相关：非演示客户、个人客户详情、企业客户详情。
- 管理员相关：普通管理员和孤立角色绑定。
- 审计相关：`setting.dblog.models` 配置模型对应的审计日志表。

基础数据阶段固定创建：

- 20 个管理员。
- 20 个车型。
- 500 台车辆。
- 5 个门店。
- 20 个客户，并为个人客户补充个人详情。

每个月生成的业务数据数量由 `setting.gen.factor` 控制。

## 合同与关联数据规则

旧版先由 `SaleContract::factory()` 直接创建合同，再根据合同 factory 随机出的租赁类型和状态补充关联数据。

主循环内的业务顺序为：

```text
创建合同
  -> 创建付款
  -> 更新车辆状态
  -> 创建临时换车
  -> 创建验车和行程
  -> 创建维修保养和违章
  -> 创建结算
  -> 创建保险
  -> 创建年检
```

关键规则：

- 长租合同通过 `SaleContractController::callPaymentsOption()` 生成付款计划，再直接创建付款。
- 短租合同在已签约及之后状态下，按 `PPtId::getFeeTypes(ScRentalType::SHORT_TERM)` 随机创建付款。
- 车辆状态在 `switch ($saleContract->sc_status)` 中按合同状态更新。
- 待签约合同一定会创建临时换车记录。
- 已签约、已完成、提前解约合同会创建两组发车验车、退车验车和行程。
- 已签约及之后合同会创建维修、保养、手工违章和普通违章。
- 已完成、提前解约合同会创建结算。
- 每个循环都会创建保险和年检。

旧版更依赖各个 factory 的默认随机字段。合同租期、金额、状态、关键时间和部分关联数据的业务一致性主要由 factory 和随机结果共同决定。

## 为什么阅读困难

旧版阅读困难主要来自代码组织方式：

- `handle()` 同时承担命令编排、清理数据、基础数据创建和业务场景生成。
- 按月循环内包含过多业务分支，阅读时很难快速区分主流程和细节规则。
- 合同状态判断分散在付款、车辆状态、换车、验车、结算等多处。
- 付款、车辆状态、验车、维保、违章、保险和年检都在同一层级串行展开。
- 业务概率、时间线和金额规则大多隐藏在 factory 默认值或就地随机逻辑里。
- 想调整某个业务场景时，需要先确认它是否依赖前面某个随机生成字段。

这些问题不一定代表旧版行为错误，但会增加维护者理解和修改命令的成本。

## 阅读代码建议

阅读旧版时建议按业务块定位，而不是从头逐行读完整个 `handle()`：

1. 先看数据清理段，确认命令会删除哪些数据。
2. 再看基础数据创建段，确认车辆、客户和门店来源。
3. 从 `SaleContract::factory()->...->create()` 开始看每月业务循环。
4. 按合同状态搜索 `ScStatus::`，理解不同状态会触发哪些关联数据。
5. 按模型名搜索 `Payment`、`VehicleInspection`、`VehicleUsage`、`VehicleRepair`、`VehicleMaintenance`、`VehicleViolation`、`SaleSettlement`、`VehicleInsurance`、`VehicleSchedule`。
6. 如需确认默认字段，继续跳到对应 factory。

旧版没有单独的业务入口方法。理解整段逻辑时，可以把“创建合同后的所有代码”看作一个隐式场景生成器。

## 注意事项

- 本文不替代数据库 schema 文档。
- 本文不完整解释每个 factory 字段。
- mock 数据只用于理解和构造开发演示数据，不承诺适合生产数据或稳定测试断言。
- 旧版说明用于理解修改前结构，不代表当前工作区仍按旧版执行。
- 验证文档或代码结构时，不需要运行 `_dev:mock-data:generate`；该命令会删除并重建演示数据。
